# Commit 1 Reflection notes

In this commit, I learned how `TcpListener::bind("127.0.0.1:7878")` is used to make the program listen for TCP connections on the local machine at port 7878. The `incoming()` method returns an iterator of incoming connection attempts, so the server can continuously accept requests as long as it is running. When I open the browser and access `http://127.0.0.1:7878`, the browser creates a TCP connection to the server, which is why the terminal prints `"Connection established!"`. Nothing appears in the browser yet because at this stage the server only accepts the connection and prints to the terminal, but it does not send any HTTP response back. I also noticed that using `_stream` avoids the unused variable warning because Rust understands that the variable is intentionally not being used. From this step, I learned the difference between merely accepting a TCP connection and actually responding to an HTTP request.

# Commit 2 Reflection notes

In this commit, I learned that `BufReader` is useful for wrapping the TCP stream so input can be read more conveniently line by line instead of raw byte by byte. The `.lines()` method reads the incoming request as separate text lines, which makes it easier to inspect the structure of the HTTP request sent by the browser. I also learned that an HTTP response must contain a status line, headers, and a body, because the browser needs all of these parts to understand whether the request succeeded and how to display the returned content. The `Content-Length` header is especially important because it tells the browser how many bytes of content are being sent, preventing incomplete or hanging responses. The file `hello.html` must be in the project root because the program reads it using a relative path, so Rust looks for it in the current working directory when `cargo run` is executed. Compared to commit 1, this version no longer just accepts connections, but now actually sends a valid HTTP response so the browser can render an HTML page.

# Commit 3 Reflection notes

In this commit, I learned that `request_line` refers to the first line of the HTTP request, such as `GET / HTTP/1.1`, and this line is enough for simple routing decisions in this tutorial. Matching `"GET / HTTP/1.1"` allows the server to detect when the user is requesting the home page and return the correct `hello.html` response. If the request line does not match a known route, the server returns a `404 NOT FOUND` status and serves `404.html`, which simulates basic error-page behavior like real web servers. I found that refactoring the logic into `(status_line, filename)` is cleaner because it separates the decision of *what response to send* from the repeated code that reads the file and writes the response to the stream. Browser behavior also changes here because now visiting an unknown path such as `/bad` no longer shows the normal page, but instead shows a dedicated not-found page. From this step, I learned the basic idea of routing and how a server can selectively respond based on the request path.

# Commit 4 Reflection notes

In this commit, I learned that the `/sleep` route delays for 10 seconds because the code explicitly calls `thread::sleep(Duration::from_secs(10))` before sending the response. Since the server is still single-threaded, it can only process one connection at a time, so while it is sleeping for one request, all other incoming requests must wait. When I tested with two browser tabs, one accessing `/sleep` and the other accessing `/`, I observed that the second tab was also delayed even though it was requesting the normal page. This demonstrates the weakness of a single-threaded server, because one slow request can block every other user from being served promptly. This is bad for many users because in a real application multiple clients may connect at the same time, and slow handling for one client would reduce responsiveness for all others. From this milestone, I learned why concurrency is important in server design and why handling requests sequentially does not scale well.

# Commit 5 Reflection notes

In this commit, I learned that a `ThreadPool` is a structure that manages a fixed set of worker threads which can repeatedly execute incoming jobs. Using a fixed number of workers is better than spawning unlimited threads for every request because it prevents excessive thread creation overhead and avoids wasting system resources under heavy load. I also learned the role of `mpsc`, `Arc`, and `Mutex` in Rust: `mpsc` is used as the channel for sending jobs from the main thread to worker threads, `Arc` allows shared ownership of the receiver across workers, and `Mutex` ensures only one worker accesses the shared receiver safely at a time. Jobs are sent to workers by boxing closures and transmitting them through the channel, after which an available worker receives and executes the job. Multithreading improves the blocking issue from the previous milestone because one worker can be busy handling `/sleep` while other workers continue serving normal requests. From this step, I learned how Rust can implement safe concurrency and how a thread pool makes a web server more responsive and practical.
